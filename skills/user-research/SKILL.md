# user-research: ユーザーリサーチ設計

## 概要

「自分が欲しいから作る」罠を回避し、実際のユーザーの課題を深く理解するための
リサーチ設計を行います。ペルソナ・インタビューガイド・仮説検証フレームを生成します。

## 「自分が欲しいから作る」罠とは

多くの失敗プロダクトは「作りたいものを作った」結果です。
このスキルは以下の問いに答えることで、その罠を回避します:

- 「誰のために作るのか」を具体化する
- 「なぜその人が必要とするのか」を検証する
- 「今どう解決しているか」を理解する

## ワークフロー

### Step 1: ターゲットユーザーのヒアリング

以下をユーザーに確認します:

```
1. このサービスを使ってほしい具体的な人物を実名で3人挙げてください。
   （実名が難しい場合は「30代女性・フリーランスデザイナー・東京在住」など詳細に）

2. その3人は今、この問題をどうやって解決していますか？
   （具体的なツール名・方法を教えてください）

3. その解決方法の何が不満ですか？
   （「遅い」「高い」「使いにくい」など具体的に）

4. あなたは過去6ヶ月以内に、その3人のうち誰かと話しましたか？
   → Yes: その会話で何を聞きましたか？
   → No: まずインタビューすることを強くお勧めします
```

3人挙げられない場合は以下を案内:
```
⚠️ ターゲットユーザーがまだ曖昧な状態です。
「誰でも使える」プロダクトは「誰にも刺さらない」リスクがあります。
まず特定の1人のユーザー像を具体化してから進めることをお勧めします。
```

### Step 2: ペルソナ生成

`docs/personas.md` を生成:

```markdown
# ユーザーペルソナ

## プライマリーペルソナ: [名前]

### 基本情報
- 年齢・職業・居住地・家族構成
- 技術リテラシー（1〜5段階）
- 月収・可処分所得の目安

### 課題と行動
- 抱えている課題（具体的に3つ）
- 現在の解決方法
- 現在の解決方法の不満点

### ゴールとモチベーション
- このサービスで達成したいこと
- 使い始めるトリガー（どんな出来事がきっかけか）
- 使い続けるモチベーション

### デザイン嗜好プロファイル
- 性格タイプ: [感覚派/合理派/創造派/効率派]
- UIの好み: [ミニマル/リッチ/データ密度高/ビジュアル重視]
- 色彩の印象: [温かみ/クール/ポップ/落ち着き]
- 参考にしていそうなサービス: [具体名3つ]
- 嫌いそうなUI要素: [具体的に]

### 引用（仮説）
「[このユーザーが言いそうなセリフを1文で]」

---

## セカンダリーペルソナ: [名前]
（同様の構成 ※デザイン嗜好プロファイルも含む）
```

### Step 3: インタビューガイド生成

`docs/interview-guide.md` を生成:

```markdown
# ユーザーインタビューガイド

## インタビューの目的
[仮説を検証するための具体的な目的]

## 事前準備
- 対象者: [ペルソナ名] に近い人物を3〜5名
- 時間: 30〜45分
- 形式: Zoom / 対面 / 電話
- 録音許可を事前に確認

## 質問リスト

### 導入（5分）
1. 「今日はお時間いただいてありがとうございます。
   まず、[課題領域] について日頃どんなふうに取り組んでいるか教えてください」

### 現状把握（10分）
2. 「[課題] を解決するために、今はどんな方法を使っていますか？」
3. 「そのやり方で、一番困っているのはどこですか？」
4. 「もし理想のツールがあったとしたら、どんな機能が必要ですか？」

### ペイン深掘り（10分）
5. 「最後に [課題] で本当に困った出来事を教えてください」
   （具体的なエピソードを引き出す）
6. 「そのとき、どう解決しましたか？」
7. 「その解決策は完璧でしたか？何が足りませんでしたか？」

### ゲイン確認（5分）
8. 「もしこの問題が完全に解決されたら、何が変わりますか？」
9. 「そのためにいくらなら払いますか？」

### クロージング（5分）
10. 「他に聞いておくべきことはありますか？」
11. 「同じ課題を抱えている知人を紹介してもらえますか？」

## 禁止事項
- ❌ 「このサービスについてどう思いますか？」（アイデアを売り込まない）
- ❌ 誘導尋問（「〜だと思いますよね？」）
- ❌ 沈黙を埋めようとする（沈黙は深く考えているサイン）

## インタビュー後のメモ
- 一番印象に残った発言:
- 仮説が検証された点:
- 仮説が覆された点:
- 次のインタビューで深掘りしたい点:
```

### Step 4: 仮説検証フレーム生成

`docs/hypothesis.md` を生成:

```markdown
# 仮説検証フレーム

## コア仮説

**ユーザー仮説**:
[ターゲットユーザー] は [課題] を抱えており、
[現在の解決策] では [問題点] があるため、
[提供価値] があるサービスに [支払い意欲/利用意欲] がある。

## 検証方法

| 仮説 | 検証方法 | 成功基準 | 期限 |
|------|----------|----------|------|
| ユーザーは課題を持つ | インタビュー5名 | 4名以上が同意 | 1週間 |
| 現在の解決策に不満 | インタビュー | 具体的不満を語れる | 1週間 |
| 支払い意欲がある | 仮価格提示 | 3名以上が「払う」 | 2週間 |
| 実際に使う | LP + 事前登録 | 50名の登録 | 3週間 |

## ピボット判断基準

以下の場合は仮説を見直す:
- インタビュー5名中3名以上が課題を否定した
- 「払う」と答えた人が0名だった
- 「今の解決策で十分」という回答が多かった
```

### 出力ファイル

- `docs/personas.md` - ユーザーペルソナ
- `docs/interview-guide.md` - インタビューガイド
- `docs/hypothesis.md` - 仮説検証フレーム

## 品質チェック

- [ ] ペルソナが実名または詳細な人物像に基づいているか
- [ ] インタビューガイドがオープンクエスチョン中心になっているか
- [ ] 仮説の検証方法と成功基準が具体的か
- [ ] 「作りたいものを売り込む」質問が含まれていないか
- [ ] ピボット判断基準が記載されているか
