# JTBD Switch Interview 質問テンプレート

## 概要

Jobs-to-be-Done（JTBD）フレームワークは、ユーザーが「なぜ」製品を切り替えるかを
4つの力で分析するインタビュー手法。

## JTBD ステートメント構造

```
When [状況/トリガー],
I want to [ジョブ/目的],
So I can [期待する成果].
```

### 記入例
```
When 月末に請求書が20枚以上たまったとき,
I want to まとめて自動で処理したい,
So I can 経理作業を半日から30分に短縮できる.
```

## Switch Interview の4つの力

```
         Push（不満）          Pull（魅力）
              ↓                    ↓
    [現状の解決策] ──────→ [新しい解決策]
              ↑                    ↑
         Habit（慣れ）        Anxiety（不安）
```

- **Push**: 現状を離れたくなる力（不満・ペイン）
- **Pull**: 新しい解決策に惹かれる力（期待・ゲイン）
- **Anxiety**: 切り替えを妨げる力（リスク・不確実性）
- **Habit**: 現状に留まろうとする力（慣れ・惰性）

Push + Pull > Anxiety + Habit のとき、ユーザーはスイッチする。

## Switch Interview 質問テンプレート

### フェーズ1: タイムライン確認（10分）

最初に「いつ」「何がきっかけで」現在の解決策を使い始めたかを聞く。

```
1. 「今使っている [ツール名/方法] を使い始めたのはいつ頃ですか？」

2. 「使い始めるきっかけは何でしたか？」
   （具体的なイベント・出来事を引き出す）

3. 「使い始める前は何を使っていましたか？」
   （さらに前の解決策を探る）
```

### フェーズ2: Push の探索（10分）

```
4. 「今の方法で一番ストレスを感じるのはどんなときですか？」

5. 「最近、この問題で本当に困ったエピソードを教えてください」

6. 「その問題が起きたとき、どんな気持ちでしたか？」
   （感情を引き出す: イライラ、不安、焦り等）

7. 「今の方法を変えたいと思ったことはありますか？
    あるなら、そのきっかけは何でしたか？」
```

### フェーズ3: Pull の探索（5分）

```
8. 「もし理想の解決策があるとしたら、どんなものですか？」

9. 「他のツール/方法を検討したことはありますか？
    何に惹かれましたか？」

10. 「この問題が完全に解決されたら、何が変わりますか？」
    （時間の節約？ストレスの軽減？売上向上？）
```

### フェーズ4: Anxiety の探索（5分）

```
11. 「新しいツールに切り替えるとしたら、何が心配ですか？」

12. 「過去にツールの切り替えで失敗した経験はありますか？」

13. 「データの移行や学習コストについて、どう感じますか？」
```

### フェーズ5: Habit の探索（5分）

```
14. 「今の方法の良いところは何ですか？」
    （現状に留まる理由を理解する）

15. 「チームや周囲の人も同じ方法を使っていますか？」
    （ネットワーク効果・慣性の強さ）

16. 「正直なところ、今の方法がまあまあ使えているから
     変える必要はないと思いますか？」
```

### フェーズ6: コミットメント確認（5分）

```
17. 「もしこの問題を解決するサービスのベータテストに
     参加してもらえますか？（週1回30分のフィードバック）」

18. 「同じ悩みを持っている知人を1人紹介してもらえますか？」
```

## インタビュー後の分析テンプレート

| 力 | 強さ(1-5) | 主な発言・エビデンス |
|----|-----------|-------------------|
| Push（不満） | | |
| Pull（魅力） | | |
| Anxiety（不安） | | |
| Habit（慣れ） | | |

**スイッチ可能性**: Push + Pull = __ vs Anxiety + Habit = __

**判定**:
- スイッチ可能性が高い（差が+3以上）→ 有望なターゲット
- 拮抗している（差が-2〜+2）→ Anxiety/Habit を下げる施策が必要
- スイッチ可能性が低い（差が-3以下）→ 別のターゲットを検討

## JTBD ジョブマップ

ユーザーのジョブを8ステップに分解して機会を発見する:

1. **定義する**: ジョブの目標を決める
2. **見つける**: 必要な情報・リソースを探す
3. **準備する**: ジョブを実行する環境を整える
4. **確認する**: 準備が整ったか確認する
5. **実行する**: ジョブのコアタスクを行う
6. **監視する**: 実行中に問題がないか確認する
7. **修正する**: 問題が発生したら対処する
8. **完了する**: ジョブを終了・後片付けする

各ステップで「満足度が低い & 重要度が高い」箇所がイノベーション機会。
