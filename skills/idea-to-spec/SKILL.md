# idea-to-spec: アイデア → 要件定義

## ワークフロー

### Step 1: ヒアリング

`docs/market-research.md` が存在する場合は読み込み、競合情報を活用してヒアリング項目を効率化する（既知の情報は確認のみで済ませる）。

以下の項目をユーザーに確認します（不明な場合のみ質問）:

1. **解決する課題**: 誰のどんな問題を解決するか（1文で）
2. **ターゲットユーザー**: 主要ユーザー像（具体的に）
3. **MVP機能**: 最初のバージョンで必須な機能（最大5個）
4. **マネタイズ方針**: 無料/フリーミアム/有料/広告など
5. **成功指標**: 3ヶ月後にどうなっていれば成功か
6. **既存代替手段**: 現在、この課題をどう解決していますか？（既存代替手段の確認）
7. **ペインの具体化**: その解決策の何が不満ですか？（ペインの具体化）
8. **ゲインの確認**: 理想の解決策を使えた場合、何が変わりますか？（ゲインの確認）
9. **APIコール・収益化**: 月間のAPIコール想定量と収益化タイムライン
10. **「自分が欲しいから作る」チェック**（必須）:
    「このサービスを使ってほしいと思う、具体的な人物を実名で3人挙げてください。
     その3人は今、この問題をどうやって解決していますか？
     その解決方法の何が不満ですか？」
    → 3人挙げられない場合: 「もう少しターゲットを具体化してから進めましょう」と案内

11. **チャネル**:
    「どのようにユーザーに最初の100人にリーチしますか？」
    （例: SNS投稿 / SEO / Product Hunt / 友人・知人 / コミュニティ / 広告）

12. **コスト試算**（重要）:
    「月間100ユーザーになったとき、インフラコストはいくらになりますか？
     Vercel Hobby は商用利用不可（Pro: $20/月）、
     Supabase Free は 7日非アクティブで一時停止します。
     スケール時の費用感を確認しておきましょう。」

13. **Vercel制限告知**（必須）: 「Vercel Hobby プランはcommercial useに利用できません。商用利用の場合はPro($20/月)が必要です。問題ありませんか？」
14. **Supabase制限告知**（必須）: 「Supabase Freeプランは500MB/2プロジェクト/7日非アクティブで一時停止の制限があります。問題ありませんか？」

既にアイデアに情報が含まれている場合は、確認のみ行います。

### Step 2: 要件定義書の生成

`docs/requirements.md` を `references/requirements-template.md` を元に生成します。
`references/evaluation-criteria.md` の5軸基準でヒアリング結果を評価し、requirements.md 内の「アイデア評価（5軸）」セクションに記載する。

### Step 3: ドメイン候補確認

プロダクト名から候補ドメインを3〜5個提示し、確認コマンドを案内:
```bash
# ドメイン空き確認
whois [domain].com
```

### 出力ファイル

- `docs/requirements.md` - 要件定義書

### Step 4: ユーザーストーリーマッピング

`references/story-mapping-template.md` に従い、MVP のスコープをストーリーマップで可視化する:

```
Backbone（ユーザー活動の大枠）
├── アクティビティ1: [例: アカウント作成]
│   ├── タスク1.1: メール登録
│   └── タスク1.2: プロフィール設定
├── アクティビティ2: [例: コンテンツ作成]
│   ├── タスク2.1: 新規作成
│   └── タスク2.2: 編集・保存
└── アクティビティ3: [例: 共有]
    ├── タスク3.1: リンク生成
    └── タスク3.2: SNS共有
```

- **Walking Skeleton**: 各アクティビティの最小タスク1つずつ → 最初のスプリント
- **MVP Release**: Walking Skeleton + 主要タスク → 初回リリース
- **Nice-to-have**: MVP 後に追加する機能

### Step 5: 非機能要件チェック

`references/non-functional-requirements-checklist.md` に従い、非機能要件を洗い出す:

| カテゴリ | 要件 | MVP基準 | 将来基準 |
|----------|------|---------|---------|
| 性能 | ページ読み込み時間 | < 3秒 | < 1.5秒 |
| 性能 | API レスポンス時間 | < 500ms | < 200ms |
| 可用性 | 稼働率 | 99% | 99.9% |
| セキュリティ | 認証方式 | Supabase Auth | + MFA |
| 拡張性 | 同時接続数 | 100 | 10,000 |

### Step 6: リスクレジスタ

主要リスクを識別・評価・対策する:

| リスク | 影響度(1-5) | 発生確率(1-5) | スコア | 対策 |
|--------|-----------|-------------|--------|------|
| PMF 未達 | 5 | 3 | 15 | 早期 PMF テスト実施 |
| 技術的負債 | 3 | 4 | 12 | CI/CD + テスト自動化 |
| コスト超過 | 4 | 2 | 8 | 月次コスト監視 |

スコア 15 以上のリスクは requirements.md の「リスクと対策」セクションに必ず記載する。

### Step 7: OMTM（One Metric That Matters）定義

プロダクトの現フェーズで最も重要な1つの指標を定義する:

```
OMTM: [指標名]
定義: [何をどう測るか]
現在値: N/A（リリース前）
目標値: [3ヶ月後の目標]
測定方法: [どのツールで計測するか]
```

フェーズ別の OMTM 例:
- **PMF検証期**: Weekly Active Users（WAU）のリテンション率
- **成長期**: 新規ユーザー獲得数（週次）
- **収益化期**: MRR（Monthly Recurring Revenue）
