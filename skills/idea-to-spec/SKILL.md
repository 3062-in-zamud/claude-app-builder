# idea-to-spec: アイデア → 要件定義

## ワークフロー

### Step 1: ヒアリング

以下の項目をユーザーに確認します（不明な場合のみ質問）:

1. **解決する課題**: 誰のどんな問題を解決するか（1文で）
2. **ターゲットユーザー**: 主要ユーザー像（具体的に）
3. **MVP機能**: 最初のバージョンで必須な機能（最大5個）
4. **マネタイズ方針**: 無料/フリーミアム/有料/広告など
5. **成功指標**: 3ヶ月後にどうなっていれば成功か
6. **既存代替手段**: 現在、この課題をどう解決していますか？（既存代替手段の確認）
7. **ペインの具体化**: その解決策の何が不満ですか？（ペインの具体化）
8. **ゲインの確認**: 理想の解決策を使えた場合、何が変わりますか？（ゲインの確認）
9. **APIコール・収益化**: 月間のAPIコール想定量と収益化タイムライン
10. **「自分が欲しいから作る」チェック**（必須）:
    「このサービスを使ってほしいと思う、具体的な人物を実名で3人挙げてください。
     その3人は今、この問題をどうやって解決していますか？
     その解決方法の何が不満ですか？」
    → 3人挙げられない場合: 「もう少しターゲットを具体化してから進めましょう」と案内

11. **チャネル**:
    「どのようにユーザーに最初の100人にリーチしますか？」
    （例: SNS投稿 / SEO / Product Hunt / 友人・知人 / コミュニティ / 広告）

12. **コスト試算**（重要）:
    「月間100ユーザーになったとき、インフラコストはいくらになりますか？
     Vercel Hobby は商用利用不可（Pro: $20/月）、
     Supabase Free は 7日非アクティブで一時停止します。
     スケール時の費用感を確認しておきましょう。」

13. **Vercel制限告知**（必須）: 「Vercel Hobby プランはcommercial useに利用できません。商用利用の場合はPro($20/月)が必要です。問題ありませんか？」
14. **Supabase制限告知**（必須）: 「Supabase Freeプランは500MB/2プロジェクト/7日非アクティブで一時停止の制限があります。問題ありませんか？」

既にアイデアに情報が含まれている場合は、確認のみ行います。

### Step 2: 要件定義書の生成

`docs/requirements.md` を `references/requirements-template.md` を元に生成します。

### Step 3: ドメイン候補確認

プロダクト名から候補ドメインを3〜5個提示し、確認コマンドを案内:
```bash
# ドメイン空き確認
whois [domain].com
```

### 出力ファイル

- `docs/requirements.md` - 要件定義書

### 品質チェック

- [ ] 解決する課題が1文で言えるか
- [ ] ターゲットユーザーが具体的か
- [ ] MVP機能が5個以内に絞れているか
- [ ] マネタイズ方針が決まっているか
- [ ] プロダクト名候補が3案以上あるか
- [ ] ビジネスデザイン3軸（DFV）が確認されているか
- [ ] 「このサービスを使ってほしい人を3人実名で挙げられているか」
- [ ] チャネル戦略が記載されているか
- [ ] スケール時のコスト試算が含まれているか
