# gdpr-compliance: GDPRコンプライアンス文書生成

## 免責事項

**このツールが生成する文書はテンプレートです。法的効力については必ずDPO（データ保護責任者）または法律専門家にレビューを依頼してください。**

## 概要

GDPR（EU一般データ保護規則）への準拠に必要なドキュメント群をテンプレートとして生成します。
データの流れを可視化し、法的根拠を整理し、データ主体の権利行使に対応するフローを設計します。

## ワークフロー

### Step 1: 入力ドキュメントの読み込み

以下のファイルを読み込み、データ処理の全体像を把握:

- `docs/requirements.md` - ビジネス要件・機能仕様
- 法務関連ドキュメント（プライバシーポリシー、利用規約等、存在する場合）

### Step 2: データマッピング

個人データの流れを可視化:

```markdown
## データマッピング

### 収集するデータ
| データカテゴリ | 具体的なデータ項目 | 収集方法 | 法的根拠 |
|--------------|-------------------|---------|---------|
| アカウント情報 | メールアドレス、氏名 | 登録フォーム | 契約 |
| 利用データ | アクセスログ、機能使用履歴 | 自動収集 | 正当利益 |
| 支払い情報 | カード情報（Stripe経由） | 決済フォーム | 契約 |
| Cookie | セッション、分析、広告 | ブラウザ | 同意 |

### データの流れ
1. **収集**: ユーザー入力 / 自動収集
2. **処理**: アプリケーションロジック
3. **保存**: Supabase (EU/US リージョン)
4. **共有**: 外部プロセッサ（Stripe, Vercel, Resend等）
5. **削除**: 保存期間満了時 / ユーザー要求時

### データ保存場所
| サービス | 保存場所 | データ種別 | DPA締結状態 |
|---------|---------|-----------|-----------|
| Supabase | AWS (リージョン指定) | ユーザーデータ全般 | 要確認 |
| Vercel | Edge Network (グローバル) | アクセスログ | 要確認 |
| Stripe | AWS (US/EU) | 支払い情報 | 締結済み（Stripe標準） |
| Resend | AWS | メールアドレス | 要確認 |
```

### Step 3: 法的根拠判定

GDPR第6条に基づき、各処理の法的根拠を判定:

```markdown
## 法的根拠一覧

| 処理活動 | 法的根拠 | 根拠の詳細 |
|---------|---------|-----------|
| アカウント作成 | 契約（Art.6(1)(b)） | サービス提供に必要 |
| メール送信（トランザクション） | 契約（Art.6(1)(b)） | サービス運用に必要 |
| メール送信（マーケティング） | 同意（Art.6(1)(a)） | オプトイン必須 |
| アクセス解析 | 正当利益（Art.6(1)(f)） | サービス改善目的 |
| Cookie（分析） | 同意（Art.6(1)(a)） | Cookie同意バナー |
| 不正利用検知 | 正当利益（Art.6(1)(f)） | セキュリティ目的 |
| 法的義務（税務記録） | 法的義務（Art.6(1)(c)） | 税法要件 |
```

### Step 4: DPA（Data Processing Agreement）テンプレート

外部プロセッサごとのDPAテンプレートを生成:

```markdown
## DPA テンプレート

### 対象プロセッサ
- Supabase: データベースホスティング
- Vercel: アプリケーションホスティング
- Stripe: 決済処理
- Resend: メール送信
- Sentry: エラー監視

### DPA 必須項目（GDPR Art.28）
1. 処理の対象・期間・性質・目的
2. 個人データの種類とデータ主体のカテゴリ
3. コントローラーの義務と権利
4. プロセッサの義務（セキュリティ措置、再委託制限等）
5. データ侵害通知義務（72時間以内）
6. 監査権
7. 契約終了時のデータ返却・削除

### 確認チェックリスト
- [ ] 各プロセッサのDPA/Data Processing Addendumを確認
- [ ] SCCs（Standard Contractual Clauses）が必要か判断（EU域外移転の場合）
- [ ] サブプロセッサ一覧を取得
```

### Step 5: DSAR（Data Subject Access Request）対応フロー

データ主体の権利行使に対応するフローを設計:

```markdown
## DSAR 対応フロー

### 対応すべき権利（GDPR Art.15-22）

#### 1. アクセス権（Art.15）
- **対応期限**: 30日以内
- **フロー**: 本人確認 → データ収集 → 機械可読形式で提供
- **実装**: `/api/gdpr/export` エンドポイント

#### 2. 訂正権（Art.16）
- **対応期限**: 30日以内
- **フロー**: 本人確認 → 対象データ特定 → 訂正 → 確認通知
- **実装**: プロフィール編集画面 + API

#### 3. 削除権（Art.17）「忘れられる権利」
- **対応期限**: 30日以内
- **フロー**: 本人確認 → 削除可否判断 → 論理削除 → 30日猶予 → 物理削除
- **実装**: data-deletionスキルと連携
- **例外**: 法的義務による保持が必要なデータ

#### 4. データポータビリティ権（Art.20）
- **対応期限**: 30日以内
- **フォーマット**: JSON / CSV（機械可読形式）
- **実装**: `/api/gdpr/export?format=json`

#### 5. 処理制限権（Art.18）
- **対応期限**: 速やかに
- **フロー**: 本人確認 → 処理停止フラグ設定 → 保存は維持
- **実装**: ユーザーテーブルに `processing_restricted` フラグ

#### 6. 自動処理に基づく決定の拒否権（Art.22）
- **対象**: 自動化された意思決定（プロファイリング含む）
- **対応**: 人間による介入の仕組みを提供
```

### Step 6: Cookie同意との整合性確認

cookie-consent スキルとの連携を確認:

- Cookie カテゴリと法的根拠の整合性
- 同意撤回時のデータ処理停止フロー
- 同意ログの保存期間

### Step 7: プライバシーインパクトアセスメント（PIA/DPIA）テンプレート

```markdown
## DPIA テンプレート（GDPR Art.35）

### 1. 処理の説明
- 処理の目的
- データの種類とカテゴリ
- データの流れ

### 2. 必要性と比例性の評価
- 処理は目的に対して必要最小限か
- より侵襲性の低い代替手段はないか

### 3. リスク評価
| リスク | 影響度 | 発生可能性 | リスクレベル | 軽減策 |
|--------|--------|-----------|-------------|--------|
| データ漏洩 | 高 | 中 | 高 | 暗号化、アクセス制御 |
| 不正アクセス | 高 | 低 | 中 | MFA、監査ログ |
| データ損失 | 中 | 低 | 低 | バックアップ |

### 4. 軽減策
[リスクごとの対策を記載]

### 5. DPO/法務レビュー
- [ ] DPOによるレビュー完了
- [ ] 必要に応じて監督機関への事前相談
```

## 出力ファイル

- `docs/gdpr-compliance.md` - GDPRコンプライアンス概要（法的根拠一覧、DPAチェックリスト、DPIA含む）
- `docs/data-mapping.md` - データマッピング（個人データの流れの可視化）
- `docs/dsar-procedure.md` - DSAR対応手順書（各権利の対応フロー）
