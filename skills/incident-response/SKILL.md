# incident-response: プロダクション・インシデント対応計画

## 概要

プロダクション環境で発生するインシデントに対して、迅速かつ体系的に対応するための計画・テンプレート・ランブックを生成する。ソロ開発者〜小規模チームを想定し、Vercel + Supabase スタックに最適化した実践的な対応手順を提供する。

## ワークフロー

### Step 1: プロジェクトコンテキスト読み込み

以下のファイルが存在すれば読み込み、プロジェクトのインフラ構成を把握する:

- `docs/monitoring-setup.md`
- `docs/deploy-setup.md`
- `docs/tech-stack.md`
- `docs/architecture.md`

存在しないファイルはスキップし、一般的なVercel + Supabaseスタックを前提に進める。

### Step 2: 重要度分類（Severity Levels）定義

以下の4段階の重要度を定義し、それぞれの判断基準・対応時間・通知先を明記する:

#### P1 (Critical) - サービス全停止・データ損失
- **判断基準**: 全ユーザーがサービス利用不可、データの損失・破損が発生または発生のリスクが高い
- **対応開始**: 即時（検知から15分以内）
- **通知**: 即時通知（Slack/Discord/PagerDuty等）
- **例**: データベース接続不能、認証サービス全停止、決済処理の異常、データ破損

#### P2 (Major) - 主要機能停止・重大パフォーマンス劣化
- **判断基準**: 主要機能が利用不可、または深刻なパフォーマンス劣化（レスポンス5秒超）
- **対応開始**: 1時間以内
- **通知**: 1時間以内に通知
- **例**: 検索機能停止、決済の一部失敗、API応答の大幅遅延

#### P3 (Minor) - 一部機能の問題・回避策あり
- **判断基準**: 一部機能に問題があるが回避策が存在する、非クリティカルな機能の停止
- **対応開始**: 営業時間内（24時間以内）
- **通知**: 日次レポートに含める
- **例**: メール通知の遅延、管理画面の表示崩れ、非重要APIのエラー率上昇

#### P4 (Low) - 軽微な問題・UX改善
- **判断基準**: 軽微なバグ、UXの改善提案、パフォーマンスの微小劣化
- **対応開始**: 次スプリントで対応
- **通知**: バックログに追加
- **例**: ボタンの配置ずれ、文言の誤り、エッジケースでの表示バグ

### Step 3: エスカレーションフロー設計

ソロ開発者を前提としたエスカレーションフローを設計する:

```
検知（監視アラート / ユーザー報告）
  ↓
重要度判定（P1-P4）
  ↓
┌─ P1/P2 → 即時対応開始
│   ├─ 自力解決可能 → 対応・復旧・ポストモーテム
│   └─ 自力解決困難 →
│       ├─ Vercel問題 → Vercel Status / Support
│       ├─ Supabase問題 → Supabase Status / Support
│       ├─ Stripe問題 → Stripe Status / Support
│       └─ 不明 → コミュニティ（Discord/Stack Overflow）/ 外部コンサル
│
└─ P3/P4 → バックログ登録 → 通常開発フローで対応
```

外部サポート連絡先リストを含める:
- 各SaaSベンダーのステータスページURL
- サポートチケット作成URL
- コミュニティフォーラムURL

### Step 4: ロールバック判断フローチャート

デプロイ起因のインシデントに対するロールバック判断フローを作成する:

```
インシデント発生
  ↓
直近のデプロイが原因か？
  ├─ Yes →
  │   ├─ Vercelデプロイ起因？
  │   │   ├─ Yes → Vercel Dashboardから前バージョンにInstant Rollback
  │   │   └─ No ↓
  │   ├─ Supabase Migration起因？
  │   │   ├─ Yes → マイグレーション巻き戻し検討
  │   │   │   ├─ 破壊的変更あり？ → 慎重に手動ロールバック（データバックアップ確認）
  │   │   │   └─ 非破壊的 → 逆マイグレーション実行
  │   │   └─ No ↓
  │   └─ 環境変数・設定変更起因？ → 前の設定に戻す
  │
  └─ No → 根本原因調査続行
```

各ロールバック手順の具体的なコマンド・手順を記載する:
- `vercel rollback` コマンドまたはDashboard操作手順
- Supabaseマイグレーションの巻き戻し手順
- 環境変数の復元手順

以下のセクションを含むポストモーテムテンプレートを作成する:

```markdown

## Timeline（時系列）
```

### Step 6: ステータスページ設定ガイド

無料〜低コストのステータスページツールの比較と設定ガイドを作成する:

- **Instatus** (Free tier): 設定手順、コンポーネント定義、自動通知設定
- **Cachet** (Self-hosted, OSS): GitHub Pagesでの無料ホスティング方法
- **GitHub Status** (リポジトリベース): GitHub Issuesを使った簡易ステータス管理
- **Betteruptime** (Free tier): 死活監視 + ステータスページ一体型

推奨コンポーネント構成:
- Webアプリケーション
- API
- データベース
- 認証
- 決済
- メール通知

### Step 7: インシデント対応ランブック（Runbook）テンプレート

よくあるインシデントシナリオごとのランブックを作成する:

1. **データベース接続エラー**
   - 確認項目: Supabase Status、コネクション数、Pooler設定
   - 対応手順: 接続プーリング再起動、フェイルオーバー確認

2. **デプロイ失敗 / ビルドエラー**
   - 確認項目: Vercelビルドログ、環境変数、依存関係
   - 対応手順: ロールバック、ローカル再現、修正デプロイ

3. **認証エラー多発**
   - 確認項目: Supabase Auth Status、JWT設定、セッション管理
   - 対応手順: トークンリフレッシュ、Auth設定確認

4. **決済処理エラー**
   - 確認項目: Stripe Status、Webhook配信状況、ログ確認
   - 対応手順: Webhook再送、手動処理、ユーザー通知

5. **パフォーマンス劣化**
   - 確認項目: Vercel Analytics、Supabaseメトリクス、外部監視
   - 対応手順: スロークエリ特定、キャッシュ確認、スケールアップ

6. **セキュリティインシデント**
   - 確認項目: 不正アクセスログ、データ漏洩確認、影響範囲
   - 対応手順: アクセス遮断、キーローテーション、ユーザー通知

各ランブックには以下を含める:
- 検知方法（どのアラートで気づくか）
- 初動対応（最初の5分で何をするか）
- 調査手順（ログの見方、確認コマンド）
- 復旧手順（具体的なステップ）
- 事後対応（ポストモーテム実施判断）
