# Go/No-Go 判断基準

## 概要

リリース前の最終判断を構造化し、感情的・政治的なリリース判断を排除する。

## 判断基準テーブル

### 必須条件（1つでも No なら No-Go）

| # | 条件 | 確認方法 | 結果 |
|---|------|---------|------|
| 1 | CRITICAL チェック項目が全て ✅ | release-checklist 参照 | Go / No-Go |
| 2 | 本番ビルドが成功する | `npm run build` | Go / No-Go |
| 3 | 本番環境の環境変数が全て設定済み | Vercel Dashboard 確認 | Go / No-Go |
| 4 | DB マイグレーションが本番に適用済み | `supabase db remote status` | Go / No-Go |
| 5 | セキュリティレビュー完了 | security-hardening 実行済み | Go / No-Go |

### 推奨条件（Go 判断の参考）

| # | 条件 | 基準 | 結果 | 許容 |
|---|------|------|------|------|
| 6 | HIGH チェック項目 | 90% 以上 ✅ | ○/△/× | △ まで許容 |
| 7 | テストカバレッジ | 80% 以上 | [%] | 70% まで許容 |
| 8 | Lighthouse Performance | 90+ | [点] | 80+ まで許容 |
| 9 | Lighthouse Accessibility | 95+ | [点] | 90+ まで許容 |
| 10 | npm audit HIGH 脆弱性 | 0件 | [件] | 0件のみ |

## 判断フロー

```
必須条件を全てチェック
  │
  ├── 1つでも No-Go → ❌ No-Go（修正してから再判断）
  │
  └── 全て Go → 推奨条件をチェック
                    │
                    ├── 全て基準内 → ✅ Go（リリース実行）
                    │
                    ├── 一部基準外（許容範囲内）
                    │    → ⚠️ Conditional Go
                    │    → 基準外項目を Issue 登録
                    │    → リリース後1週間以内に対応
                    │
                    └── 許容範囲外
                         → ❌ No-Go（修正してから再判断）
```

## Conditional Go のルール

Conditional Go（条件付きリリース）が許容されるのは以下の場合のみ:

1. 未達項目がユーザー体験に直接影響しない
2. 未達項目の修正計画と期限が明確
3. ロールバック手順が確認済み
4. モニタリング体制が整っている

Conditional Go は使いすぎると品質低下を招くため、
連続して Conditional Go になる場合はプロセスの見直しが必要。
