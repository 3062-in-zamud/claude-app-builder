# retention-strategy: リテンション戦略

## 概要

「新規獲得の5倍のコストがかかる」と言われる顧客維持。コホート分析でリテンションの実態を把握し、
チャーン予測シグナルを定義し、Re-engagementメールとHabit Loop設計で継続利用を促進します。

## ワークフロー

### Step 1: 入力ドキュメントの読み込み

以下のファイルを読み込み、現状のユーザー行動と計測基盤を把握:

- `docs/requirements.md` - ビジネス要件・ターゲットユーザー
- `docs/analytics-plan.md` - 計測イベント設計
- `docs/feedback-strategy.md` - フィードバック戦略（存在する場合）

### Step 2: コホート分析設計

週次・月次のリテンションカーブを計測するためのSQLクエリテンプレートを設計:

```sql
-- 週次コホートリテンション分析
WITH cohort AS (
  SELECT
    user_id,
    DATE_TRUNC('week', created_at) AS cohort_week
  FROM users
),
activity AS (
  SELECT
    user_id,
    DATE_TRUNC('week', event_at) AS activity_week
  FROM user_events
  WHERE event_name = 'session_started'
)
SELECT
  c.cohort_week,
  COUNT(DISTINCT c.user_id) AS cohort_size,
  COUNT(DISTINCT CASE
    WHEN a.activity_week = c.cohort_week + INTERVAL '1 week'
    THEN a.user_id END) AS week_1,
  COUNT(DISTINCT CASE
    WHEN a.activity_week = c.cohort_week + INTERVAL '2 weeks'
    THEN a.user_id END) AS week_2,
  -- ... Week N まで
FROM cohort c
LEFT JOIN activity a ON c.user_id = a.user_id
GROUP BY c.cohort_week
ORDER BY c.cohort_week;
```

```markdown
## コホート分析ダッシュボード設計

### リテンションカーブ
| コホート | Size | W1 | W2 | W3 | W4 | W8 | W12 |
|---------|------|----|----|----|----|----|----|
| Week 1  | 100  | 40%| 25%| 20%| 18%| 15%| 12%|

### ベンチマーク（SaaS業界）
- Day 1 リテンション: 40-60%
- Week 1 リテンション: 20-35%
- Month 1 リテンション: 15-25%
- Month 3 リテンション: 10-15%

### アクション閾値
- W1 < 20%: オンボーディングに問題あり → 改善優先
- W4 < 10%: プロダクト価値の訴求不足 → Aha Moment 再設計
- W12 < 5%: プロダクトマーケットフィットの再検証が必要
```

### Step 3: チャーン予測シグナル定義

ユーザーの離脱を予測するためのシグナルを定義:

```markdown
## チャーン予測シグナル

### Early Warning Signs（早期警告）
| シグナル | 閾値 | 重み | 計測方法 |
|---------|------|------|---------|
| 最終ログインからの経過日数 | > 7日 | 高 | `last_sign_in_at` |
| 週間セッション数の減少 | 前週比 50%以下 | 高 | `session_started` イベント |
| コア機能の使用頻度低下 | 前週比 30%以下 | 中 | `feature_used` イベント |
| サポート問い合わせ未解決 | > 3件 | 中 | サポートチケットDB |
| ダウングレード検討 | プラン変更ページ訪問 | 高 | `page_viewed` イベント |
| エクスポート実行 | データエクスポート | 高 | `data_exported` イベント |

### チャーンスコア計算
各シグナルの重みを合算してリスクスコアを算出:
- 0-30: 低リスク（通常モニタリング）
- 31-60: 中リスク（Re-engagementメール送信）
- 61-100: 高リスク（カスタマーサクセス介入）

### リスク別アクション
| リスクレベル | アクション | タイミング |
|------------|-----------|-----------|
| 低 | 定期ニュースレター | 週次 |
| 中 | Re-engagementメール | シグナル検知時 |
| 高 | パーソナライズド介入 | 即時 |
| 離脱済み | Winbackキャンペーン | 離脱後14日 |
```

### Step 4: Re-engagementメール設計

トリガーベースのメールシーケンスを設計:

```markdown
## Re-engagementメールシーケンス

### 7日無活動トリガー
- **件名例**: 「[サービス名]で[具体的な価値]が待っています」
- **内容**: 最近の新機能紹介、ユーザーの未完了タスクのリマインド
- **CTA**: 「続きを始める」

### 14日無活動トリガー
- **件名例**: 「[ユーザー名]さん、お元気ですか?」
- **内容**: 他ユーザーの成功事例、Tips紹介
- **CTA**: 「新しい使い方を見る」

### 30日無活動トリガー
- **件名例**: 「フィードバックを聞かせてください」
- **内容**: 簡単なアンケート（なぜ使わなくなったか）
- **CTA**: 「1分で回答する」
- **インセンティブ**: 回答者に[特典]を付与

### メール設計原則
- パーソナライゼーション（ユーザー名、利用データ）
- 1メール1CTA（選択肢を増やさない）
- 配信停止リンク必須
- 送信頻度制限（週1回まで）
- A/Bテスト（件名、CTA、送信時間）
```

### Step 5: Habit Loop設計

Nir Eyal の Hook Model に基づくHabit Loop設計:

```markdown
## Habit Loop設計（Hook Model）

### 1. Trigger（きっかけ）
#### External Triggers（外部トリガー）
- メール通知（週次ダイジェスト、アクティビティ通知）
- プッシュ通知（重要な更新のみ）
- SNS連携（共有された時の通知）

#### Internal Triggers（内部トリガー）
- [ユーザーが感じる課題/欲求]が発生した時
- 例: 「あのデータを確認したい」「進捗を記録したい」

### 2. Action（行動）
- アクションの障壁を最小化
- ログイン不要で価値を体験できる導線
- モバイル対応でいつでもアクセス可能

### 3. Variable Reward（変動報酬）
- **Tribe（社会的報酬）**: 他ユーザーからのフィードバック、ランキング
- **Hunt（情報の報酬）**: 新しいインサイト、パーソナライズされたレコメンド
- **Self（自己達成の報酬）**: 進捗の可視化、マイルストーン達成

### 4. Investment（投資）
- ユーザーがサービスに価値を蓄積する仕組み
- データ入力、カスタマイズ、コネクション構築
- 投資が増えるほど離脱コストが上がる
```

### Step 6: Feature Adoption 追跡

```markdown
## Feature Adoption 追跡

### 新機能ローンチ計測
| 指標 | 計測方法 | 目標 |
|------|---------|------|
| 認知率 | 機能告知の表示回数 / DAU | > 80% |
| 試用率 | 初回使用者数 / 認知者数 | > 30% |
| 定着率 | 2回以上使用者 / 試用者数 | > 50% |

### 機能別利用率ダッシュボード
- 日次/週次の機能別利用者数
- パワーユーザー vs. ライトユーザーの利用パターン差
- 機能間の相関（機能Aを使う人は機能Bも使う傾向）
```

### Step 7: チャーンインタビュー計画

```markdown
## チャーンインタビュー

### 対象者選定
- 過去30日以内に解約したユーザー
- 有料プランからのダウングレードユーザー
- 目標: 月5件以上のインタビュー実施

### 質問テンプレート
1. [サービス名]を使い始めたきっかけは何でしたか?
2. どのような問題を解決しようとしていましたか?
3. 実際に使ってみて、期待通りでしたか?
4. 使わなくなった/解約した主な理由は何ですか?
5. 他のサービスに乗り換えましたか? どのサービスですか?
6. どうすれば使い続けていましたか?
7. [サービス名]で最も価値があったと感じる機能は?
8. 改善してほしかった点は?

### 分析フレームワーク
- **カテゴリ分類**: 価格/機能不足/UX/競合/不要になった
- **定量化**: カテゴリ別の件数と割合
- **優先度付け**: 改善インパクト x 改善容易性
```

### Step 8: Winback戦略

```markdown
## Winback戦略（離脱ユーザー復帰施策）

### タイミング
- 離脱後14日: 最初のWinbackメール
- 離脱後30日: 特別オファー付きメール
- 離脱後90日: 最終Winbackメール

### Winbackオファー
| タイミング | オファー内容 | 目標復帰率 |
|-----------|------------|-----------|
| 14日後 | 新機能/改善点の紹介 | 5% |
| 30日後 | 30%割引クーポン（期間限定） | 3% |
| 90日後 | 無料トライアル延長 | 1% |

### セグメント別アプローチ
- **価格理由の離脱**: 割引オファー、年払い提案
- **機能不足の離脱**: 要望機能リリース時に通知
- **競合乗り換え**: 差別化ポイントの訴求
- **不要になった**: 新しいユースケースの提案

### 効果計測
- Winbackメール開封率
- Winback経由の復帰率
- 復帰ユーザーのLTV（通常ユーザーと比較）
```
